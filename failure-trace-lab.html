## Root-cause chain

1. Surface: user receives (or would have received, absent blocks) manual 'open the app and tap X' instructions, confidently 'verified' claims, and raw restart-scaffold text in #shadow-hq.
2. Behavioral: post-restart, the agent defaulted to describing actions and recalling state instead of executing tools and reading files — 12 FM-012 fires and 6 grounding fires in ~110 minutes, spanning iOS and Discord, show a session-wide execution-avoidance posture, not isolated slips.
3. Contextual: the restart is the likely trigger — the '[System: Bot just restarted. Completed before restart: nothing]' scaffold appearing in outbound messages shows the agent treating injected recovery context as content, and the same context loss explains why it answered state questions from stale memory rather than re-reading state files.
4. Structural: warn-severity gates (state-assertion-grounding fired 4x without blocking) correct the symptom per-turn but don't change the generation policy; the model regenerates the same pattern seconds later (01:00:39 → 01:00:57 identical block pair), so anything below block severity functions as telemetry, not control.
5. Root: there is no post-restart re-grounding contract — nothing forces a state-file read and a tool-capability probe before the first substantive response after a restart, so every restart re-spawns a session that instructs instead of executes and asserts instead of reads until the per-turn gates grind it back into shape.

## The contract that would have caught it

**`post-restart-regrounding-gate`**

- **Trigger:** First N substantive responses after a bot-restart scaffold is injected into context (detectable from the same '[System: Bot just restarted]' token the egress guard already matches).
- **Precondition:** Before any user-facing response in the post-restart window: (a) at least one ground-truth read of the relevant state file or platform API has run this turn, and (b) if the task names a platform (ios/discord/gmail), a programmatic tool call for that platform was attempted before any response containing imperative UI verbs ('open', 'tap', 'go to').
- **Why it catches this:** Every violation cluster in this window sits downstream of the restart: the 01:00:39–01:00:57 iOS instruction pair, the four assert-from-memory warns, and the scaffold-echo egress fires all occur in the post-restart re-grounding gap. A single gate that blocks the first response until a live read + tool attempt exist would have converted 12 reactive per-turn fires into one proactive precondition, instead of letting the same policy failure re-fire across two platforms for two hours.

## Why this matters

Prompt-only guardrails would have caught none of this: the agent's instructions already say 'execute, don't instruct' and 'never assert without a read' — yet the identical violation regenerated 18 seconds after being blocked, and 12 times across the window. Deterministic contracts turned every one of those regenerations into a blocked or sanitized event with a named failure mode, timestamp, and severity, which is what makes this autopsy possible at all. The trace also shows the honest limitation: warn-severity gates are telemetry, not control — the buyer lesson is that enforcement must be block-severity at the point of egress, with layered redundant contracts, because the model will keep generating the failure no matter what the prompt says.
