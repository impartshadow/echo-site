Redact before you submit — the validator fails closed

Submissions are machine-scanned before anything is stored. If the payload contains any secret-shaped token or personal identifier — API keys, bearer tokens, JWTs, private keys, emails, phone numbers, IP addresses, SSNs, password= assignments — the whole submission is rejected and nothing is persisted. Accepted traces are stored only as sanitized fixtures, never as live customer data. Use generic labels ("agent_label": "billing-ops-agent"), not company or people names.

Format (copy/paste this shape)

{
  "schema": "shadow.failure_trace_submission.v1",
  "agent_label": "support-triage-agent",
  "events": [
    {"ts": "2026-07-01T14:02:00Z",
     "step": "agent claimed the refund was issued and closed the ticket",
     "expected": "refund API call in the trace before the completion claim",
     "observed": "no API call — completion message generated from context alone"},
    {"ts": "2026-07-01T14:09:00Z",
     "step": "operator asked for the refund receipt; agent produced a transaction id",
     "observed": "transaction id does not resolve in the payment system"}
  ]
}
  • Up to 50 events, each field ≤600 characters.
  • step is required per event; expected / observed are optional but make the autopsy sharper.
  • One failure per submission — the free tier triages one concrete failure, not a whole log dump.
